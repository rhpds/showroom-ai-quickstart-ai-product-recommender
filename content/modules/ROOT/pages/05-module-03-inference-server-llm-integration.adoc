== Exercise 1: Deploy and test the inference server with KServe

You need to understand how Kaleido's trained models are deployed for production use. The Red Hat inference server, powered by KServe, transforms static model files into scalable services that handle thousands of concurrent requests.

Understanding inference server capabilities is essential for deploying models that meet production performance and reliability requirements.

=== Why inference servers matter

ML models are **static collections of numbers** representing learned patterns. To make predictions:

1. Load model weights into memory (can be gigabytes in size)
2. Run computations over input data (matrix multiplications, activations)
3. Return predictions or embeddings

A naive approach would process one request at a time.

This would be **painfully slow** -- like waiting in a grocery checkout line behind customers with full carts when you only have a few items.

**Inference servers solve this by**:

* **Model persistence** -- load once, keep in memory, serve many requests
* **Concurrent processing** -- handle multiple requests simultaneously
* **Efficient batching** -- group requests to maximize hardware utilization
* **Auto-scaling** -- add/remove instances based on load

=== Continuous batching: The grocery store analogy

OpenShift AI's inference server uses **continuous batching** to optimize throughput:

**Traditional batching** (inefficient):

* Wait for 32 requests to accumulate
* Process all 32 together as one batch
* While processing, new requests queue up
* Result: High latency for requests that arrive just after a batch starts

**Continuous batching** (efficient):

* Process one item from each customer's cart in rotation
* As new customers arrive, add them to the rotation
* No request waits for a full batch to accumulate
* Result: Consistent low latency, high throughput

When paired with **accelerators** (NVIDIA GPUs, Intel Gaudi), continuous batching keeps hardware busy processing different requests in parallel.

=== KServe: Model serving on Kubernetes

**KServe** is a Kubernetes-native framework for model serving that:

* **Standardizes model deployment** -- consistent API across model types
* **Handles traffic routing** -- canary deployments, A/B testing
* **Auto-scales inference pods** -- based on request volume
* **Provides observability** -- metrics, logging, tracing

The Red Hat inference server integrates with KServe to provide:

* **Optimized runtimes** -- vLLM for LLMs, other runtimes for different model types
* **Quantization** -- reduce model size/memory with minimal accuracy loss
* **Multi-model serving** -- multiple models on one instance

== Exercise 2: Generate product review summaries with LLMs

Now let's explore how Kaleido uses the LLMs to generate product review summaries. This capability helps customers quickly understand product quality without reading hundreds of reviews.

Understanding LLM integration and context window management is essential for production generative AI applications.

=== The product recommender with review summaries

When customers browse products, they see:

image::figure-1-landing-page-shows-recommendations-for-the-logged-in-user.png[Landing page showing personalized product recommendations with search bar for text and image queries,link=self,window=blank,width=800,title="Landing Page with Personalized Recommendations"]

Clicking on a product shows details and reviews:

image::figure-2-product-details-page.png[Product details page showing product image, ratings, reviews, and AI Summarize button,link=self,window=blank,width=800,title="Product Details Page with Reviews"]

The **AI Summarize** button triggers review summarization:

image::figure-3-ai-generated-summary-of-reviews.png[AI-generated summary showing concise paragraph distilling user reviews with pros, cons, and recommendation,link=self,window=blank,width=800,title="AI-Generated Review Summary"]

The LLM processes reviews and generates a structured summary covering:

* **Overall sentiment** -- positive, mixed, or negative
* **Key pros** -- what customers liked
* **Key cons** -- common complaints
* **Recommendation** -- whether to buy based on review consensus

=== How review summarization works

When you click "AI Summarize":

1. Backend fetches all reviews for the product from PostgreSQL
2. Reviews are formatted into a prompt with instructions
3. Prompt is sent to LLM model via the inference server endpoint
4. LLM generates a summary following the template structure
5. Summary is returned to the frontend and displayed

**Example prompt structure**:

[source,text]
----
Summarize the following product reviews. Include:
- Overall sentiment
- Main pros (2-3 points)
- Main cons (2-3 points)
- Buying recommendation

Reviews:
1. "Great product, fast charging works perfectly. Rating: 5/5"
2. "Good quality but a bit pricey. Rating: 4/5"
3. "Charges quickly but gets warm during use. Rating: 4/5"
...

Generate a concise summary:
----

The LLM analyzes patterns across reviews and produces a coherent summary.

=== LLM context window limitations

One challenge when using LLMs is the **limited context window** -- the maximum number of tokens the model can process in one request.

**Tokens** are word fragments. For example:

* "portable" might be 1 token
* "preconfigure" might be 2 tokens: "pre" + "configure"

**Llama 3.1 8B theoretical maximum**: 128,000 tokens

**But practical limits are much lower** due to:

* **VRAM capacity** -- GPU memory is limited (24 GB on NVIDIA A10G)
* **Context rot** -- too much input text distracts the model, reducing accuracy
* **Non-uniform attention** -- models focus on the beginning and end, neglect the middle

**Real-world capacity on A10G GPU**:

* Model weights: ~17 GB
* Available for context: ~7 GB
* Approximately 14,000 tokens (about 10,500 words)

If a product has 200 reviews averaging 50 words each, that's **10,000 words** -- approaching the limit.

=== Handling products with many reviews

The product recommender uses **weighted stratified sampling** to stay within context limits:

**Algorithm**:

1. Set target review budget (e.g., 1,000 reviews)
2. Group reviews by rating (1-star, 2-star, 3-star, 4-star, 5-star)
3. Calculate weight for each group based on frequency
4. Sample reviews proportionally from each group
5. Within each group, sort by recency (favor newer reviews)
6. Send sampled reviews to LLM

**Why weighted stratified sampling?**

* **Represents rating distribution** -- if most reviews are 4-5 stars with a spike at 1-star, sample reflects this
* **Captures sentiment range** -- includes both positive and negative perspectives
* **Prioritizes recent feedback** -- newer reviews are more relevant

**Alternative approaches** (not used but worth knowing):

* **Aspect-guided summarization** -- summarize separately by product characteristics (battery life, build quality, price)
* **Question-based filtering** -- generate questions from reviews, check if summary answers them
* **Hierarchical summarization** -- summarize in batches, then summarize the summaries

== Exercise 3: Explore user registration and cold-start recommendations

Finally, let's explore how Kaleido handles **new users** who don't have interaction history yet. This is called the **cold-start problem** -- you can't recommend products based on past behavior if there is no past behavior.

Understanding cold-start strategies is essential for production recommendation systems that must serve all users, not just those with extensive history.

=== The cold-start problem

The two-tower model creates user embeddings based on:

* Preferred categories
* Interaction history (purchases, views, ratings)
* Behavioral patterns

**New users have none of this**. You could:

* **Option 1**: Run user encoder on empty features (produces poor, generic embedding)
* **Option 2**: Wait for interaction data (users see nothing until they interact)
* **Option 3**: Gather initial preferences and use popularity-based recommendations

Kaleido's product recommender uses **Option 3** with a multi-step registration process.

=== Registration workflow

**Step 1: Category selection**

image::figure-4-new-user-registration.png[New user registration showing checkboxes for product categories like Cables & Accessories and Laptop Accessories,link=self,window=blank,width=700,title="New User Registration - Category Selection"]

Users select broad categories of interest:

* Electronics
* Home & Garden
* Sports & Outdoors
* Fashion & Accessories

They can also drill down to subcategories for more precision.

**Step 2: Product selection**

image::figure-5-product-selection.png[Product selection showing grid of products with images, titles, descriptions, prices, and checkboxes for selection,link=self,window=blank,width=800,title="Product Selection for Preference Profiling"]

After selecting categories, users choose at least 10 specific products they like.

These selections serve 2 purposes:

1. **Immediate recommendations** -- show popular products in selected categories
2. **Training data** -- selections are recorded as positive interactions for next training run

=== How initial recommendations work

Until the next scheduled training run (typically nightly):

**Recommendation strategy**:

* Filter products by selected categories
* Rank by popularity (total purchases, average rating, view count)
* Return top-k products

**Why this works**:

* Popular products in relevant categories are likely to interest new users
* Provides immediate value without waiting for training
* User selections generate interaction data for better future recommendations

**After the next training run**:

* User's product selections become positive interactions
* Two-tower model creates a proper user embedding
* Recommendations switch from popularity-based to personalized embedding-based

=== Alternative cold-start approach

You could also use the **user encoder immediately** by:

* Using selected categories as user features
* Leveraging the encoder's learned patterns from similar users
* Comparing new user embedding to pre-computed product embeddings

This is more expensive (requires real-time encoding) but provides better initial recommendations than pure popularity ranking.

== Troubleshooting

**Issue**: AI Summarize button doesn't generate summary

**Solution**:

. Check if product has at least 4 reviews (minimum requirement)
. Verify Llama model deployment: `oc get inferenceservice`
. Check model endpoint is accessible and has valid authentication
. Review backend logs for LLM connection errors: `oc logs <backend-pod>`

**Issue**: Review summary seems generic or low quality

**Solution**:

. Check if enough reviews were sampled (review count in database)
. Verify weighted stratified sampling is capturing rating distribution
. Increase sample budget if context window allows
. Check prompt template for clarity and specificity

**Issue**: LLM requests timing out or failing

**Solution**:

. Check GPU availability: `oc get pod -o wide` (verify GPU nodes)
. Verify vLLM pod has sufficient memory
. Review model resource limits and requests
. Check if multiple concurrent requests are overwhelming the instance

**Issue**: Inference endpoint returns 404 or connection refused

**Solution**:

. Verify KServe route is created: `oc get route`
. Check inference service is ready: `oc get inferenceservice`
. Ensure model loaded successfully: review pod logs
. Verify authentication token if required

**Issue**: New user sees no recommendations after registration

**Solution**:

. Check if categories were properly saved to database
. Verify products exist in selected categories
. Review popularity scoring logic (ensure products have ratings/purchases)
. Check backend logs for database query errors

**Issue**: Understanding context window vs VRAM capacity

**Solution**:

. **VRAM** = total GPU memory (e.g., 24 GB on A10G)
. **Model weights** take most VRAM (17 GB for Llama 3.1 8B)
. **Remaining VRAM** available for context (7 GB → ~14k tokens)
. **Context window** = maximum tokens model can process (theoretical 128k, practical 14k)
