=== Understanding model architecture choices

When building AI applications, engineers must consider the **task** and **data type** each model supports.

**Encoder-only models** (BGE, CLIP):

* Convert inputs (text or images) to numeric representations called **embeddings**
* Embeddings are lists of numbers that capture semantic meaning
* Similar items have embeddings that are close together in vector space
* Used for search, similarity, and classification tasks

**Decoder-only models** (Llama 3.1):

* Generate new tokens sequentially in response to instructions
* Take a prompt and produce human-readable text
* Used for summarization, question answering, and content generation

**Dual-encoder models** (Two-Tower):

* Train 2 separate encoders (one for users, one for products)
* Create embeddings in a **shared space** where distance reflects interaction strength
* Enable recommendations by finding nearest product embeddings to user embeddings

=== How embeddings enable semantic search

Embeddings are the foundation of modern AI search and recommendations. Here's why they matter:

Traditional keyword search matches exact words -- searching for "thin remote controls" won't find products described as "slim remotes." Embedding models understand that these phrases mean the same thing because they learned semantic relationships during training.

The BGE-small model converts text to 384-dimensional vectors. Each dimension captures a different aspect of meaning. When you search for a product, the system:

. Converts your search text to a 384-dimensional embedding
. Compares it to pre-computed embeddings for all products in the catalog
. Returns products with the closest (most similar) embeddings

This is why the application can find relevant products even when your search terms don't exactly match the product description.

**Mathematical insight**:

Embeddings form an **embedding space** where vector arithmetic has semantic meaning. For example:

----
Vector('Trail Hiker's Backpack') + Vector('Upgrade') ≈ Vector('Mountaineer's Backpack')
----

Neural networks learn these embedding spaces through **contrastive learning** -- processing pairs of similar and dissimilar inputs until related items cluster together.

=== Why different architectures matter

You might wonder: "Why not use one model for everything?"

The architecture determines what a model can do efficiently:

* **Encoder-only models** excel at understanding and representing inputs but can't generate new content. They're fast and efficient for search.
* **Decoder-only models** excel at generating coherent text but are slower and use more resources. They're essential for summarization.
* **Dual-encoder models** learn coordinated representations for 2 different types of entities (users and products), enabling personalized recommendations.

Choosing the right architecture for each task optimizes both performance and resource usage.

== Exercise 3: Review the FEAST feature store

AI models only learn from data, so the features used during training must match those used during inference. Otherwise, accuracy suffers through a phenomenon called **training-serving skew**.

FEAST is a feature store that provides a single source of truth for feature definitions, ensuring consistency across the ML lifecycle.

=== Why feature stores matter

Consider this scenario:

During training, your recommendation model learns from product descriptions stored in a PostgreSQL database. But 3 months later, the data team migrates to a new schema and changes how descriptions are formatted. Your model still expects the old format, causing accuracy to drop.

Feature stores prevent this by:

* **Versioning features** -- changes don't break existing applications
* **Unified Python API** -- consistent access regardless of underlying storage
* **Data validation** -- ensures features match expected schemas

=== FEAST architecture in the product recommender

The product recommender uses FEAST with PostgreSQL + pgvector for feature storage:

* **Offline store** (Parquet files) -- historical data for model training
* **Online store** (PostgreSQL with pgvector) -- low-latency feature retrieval for inference

image::figure-2-sample-feast-api-usage-pattern.png[Sample FEAST API usage showing data source, feature view, and FeatureService,link=self,window=blank,width=700,title="Sample FEAST API Usage Pattern"]

The workflow is:

. Define a **data source** (Parquet file or live database)
. Create a **feature view** selecting specific columns
. Build a **FeatureService** from one or more views
. Client code retrieves features through a consistent API

=== How FEAST prevents training-serving skew

When you define features in FEAST:

* Training code reads from the offline store
* Inference code reads from the online store
* Both use the same feature definitions and transformations
* Feature versioning ensures backward compatibility

This guarantees that the model sees the same data structure in production that it saw during training.

== Troubleshooting

**Issue**: Cannot access product recommender application

**Solution**:

. Verify the URL is correct
. Check that pods are running: `oc get pods`
. Review route status: `oc get routes`
. Check application logs: `oc logs <pod-name>`

**Issue**: Search results seem irrelevant or incorrect

**Solution**:

. Embeddings may not be loaded -- check if pipeline has run
. Verify feature store contains product embeddings
. Review pipeline logs for embedding generation errors
. Confirm BGE model is deployed and accessible

**Issue**: AI summarize button doesn't work

**Solution**:

. Check if Llama model is deployed: `oc get inferenceservice`
. Verify model endpoint is accessible
. Check if product has at least 4 reviews (minimum for summarization)
. Review backend logs for LLM connection errors

**Issue**: Understanding embedding dimensions

**Solution**:

. Embedding dimensions are fixed by model architecture
. BGE-small uses 384 dimensions
. More dimensions capture more nuance but require more storage/compute
. Don't try to change dimensions -- they're learned during pre-training

**Issue**: Confused about when to use encoder vs decoder models

**Solution**:

. Need to search or compare items? → **Use encoder** (creates embeddings)
. Need to generate new text? → **Use decoder** (LLM)
. Need coordinated representations for 2 entities? → **Use dual-encoder**
