== Key takeaways

The most important concepts to remember:

. **Architecture matters for AI applications**: Different AI model types serve different purposes. Encoder-only models create embeddings for search and similarity, decoder-only models generate text, and dual-encoder models enable recommendations by learning coordinated representations. Choosing the right architecture for each task optimizes both performance and resource usage.

. **Feature consistency prevents production failures**: Training-serving skew occurs when the features used during model training don't match what the model receives in production. FEAST feature store solves this by providing a single source of truth for feature definitions, versioning, and transformations across offline (training) and online (inference) storage layers.

. **Production ML requires orchestration and serving infrastructure**: Kubeflow Pipelines automate complex training workflows by chaining Python functions into containerized components with automatic data transfer. The Red Hat inference server transforms static model files into scalable services using continuous batching to handle thousands of concurrent requests efficiently.

. **LLM integration requires context window management**: LLMs have theoretical token limits (128,000 for Llama 3.1) but practical limits are much lower due to VRAM capacity, context rot, and non-uniform attention. Production applications use techniques like weighted stratified sampling to represent data within budget while maintaining quality.

. **Cold-start strategies ensure universal coverage**: Recommendation systems must serve all users, not just those with extensive history. Multi-step registration gathers initial preferences and provides popularity-based recommendations until personalized models can be trained with accumulated interaction data.

== Next steps

Ready to continue your journey? Here are some recommended next steps:

=== Expand your OpenShift AI skills

Continue building AI applications with these advanced topics:

* **Custom serving runtimes** -- Deploy models with specialized inference servers optimized for your hardware and use case
* **Model monitoring and observability** -- Track model performance, drift, and data quality in production deployments
* **AutoML and hyperparameter tuning** -- Automate model selection and optimization with tools like Optuna integrated into Kubeflow Pipelines
* **Multi-model serving** -- Deploy multiple models on a single inference server to optimize resource utilization
* **Continuous training pipelines** -- Implement automated retraining workflows triggered by data drift detection or scheduled intervals

=== Practice projects

Put your new skills to work with these project ideas:

. **Content-based recommendation system**: Build a news article or blog post recommender using BGE embeddings and semantic search, then add user personalization with a simplified two-tower model trained on click data.

. **Chatbot with retrieval-augmented generation (RAG)**: Combine semantic search over a knowledge base with LLM generation to create a customer support chatbot that answers questions accurately with source citations.

. **Image similarity search for e-commerce**: Use CLIP embeddings to build a visual search feature that finds products similar to uploaded images, then integrate with the product recommender for hybrid text-image recommendations.

. **Review sentiment analyzer**: Train a classification model on review text and ratings, deploy it with KServe, and integrate with the review summarization feature to tag summaries with detailed sentiment breakdowns.

. **Real-time feature store with streaming data**: Extend the FEAST integration to handle real-time user interactions from Kafka or similar streaming platforms, enabling immediate recommendation updates without waiting for nightly batch jobs.
